# 处理阶段中断设计

## 设计目标（用户视角）
用户在处理阶段可以随时中断任务执行，界面立即反馈“已取消”，并安全停止正在运行的任务，不再继续新的任务，同时保留已完成结果供查看。

## 需求拆解
- 处理界面支持中断操作（快捷键）。
- 中断后停止发起新任务，并取消正在执行的任务。
- UI 状态明确显示“已取消”，并允许退出或回到浏览。

## 现状
- 执行器 `Executor` 支持 context 取消，但 UI 未提供取消入口。
- 处理阶段事件通过 channel 驱动 UI 更新。

## 方案概述
引入可取消的 context 与用户交互：
1. 在 `model` 中保存 `context.CancelFunc`。
2. Processing 状态下捕获 `esc`（或自定义键）触发取消。
3. 调用 cancel 后，执行器停止分发新任务，worker 退出。
4. UI 显示“已取消”，并允许返回或退出。

## 交互设计
- `esc`：取消处理。
- `q`：退出（保持现有行为）。

## 状态与消息
- 新增 `ProcessingCanceledMsg`（或在 ProcessingScreen 内维护 `Canceled` 标志）。
- 当取消后，UI 显示取消状态，并停止继续拉取事件。

## 结果处理
- 已完成任务结果保留。
- 未完成任务标记为 Pending 或 Canceled（可选）。

## 修改范围
- `internal/ui/model.go`：持有 cancel func，处理按键并触发取消。
- `internal/app/executor.go`：确保 ctx 取消后 worker 退出（已具备，但需验证）。
- `internal/ui/screens/processing.go`：展示取消状态与提示。

## 风险与应对
- **取消时并发任务未能立即停止**：依赖底层 ffmpeg/probe 是否响应 ctx，若不响应则可能延迟结束。
- **事件通道阻塞**：取消后需安全关闭或停止消费。

## 验证方式
- 手动：处理过程中按 `esc`，观察任务不再继续，UI 显示“已取消”。
- 结果：已完成任务保持，未完成任务不再变化。
